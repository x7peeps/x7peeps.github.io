## 二、完整工作流（5 阶段）

### Phase 1: 需求确认

1. **确认文章主题** — 明确要写什么、面向什么受众
2. **确认所属分类** — 判断放入已有分类还是新建分类
3. **确认排序权重** — 查看同分类下已有文章的 weight 值，避免冲突
4. **确认是否批量** — 如果用户要求一次写多篇，按 P1/P2/P3 分批并行执行

### Phase 2: 网络调研

1. 使用 **WebSearch** 搜索主题相关的：
   - 核心论文和学术资源（arXiv、Google Scholar）
   - 开源项目和 GitHub 仓库（star 数、活跃度）
   - 行业报告和技术博客（官方文档优先）
   - 最新进展（注意当前日期，搜索最新内容）
2. 使用 **WebFetch** 获取关键页面的详细内容
3. **整理调研素材**，形成结构化大纲：
   - 核心概念定义
   - 技术演进脉络
   - 主流方案对比
   - 关键代码/配置示例
   - 参考资源列表

### Phase 3: 文章撰写

按照下方「写作规范」和「Frontmatter 规范」生成完整文章。

### Phase 4: 构建验证

```bash
cd "hugo-src" && hugo
```

验证要点：
1. **构建成功** — 无 ERROR 输出（WARNING 可忽略）
2. **页面数增长** — 确认总页面数合理增加（新增 1 篇文章应增加约 3-5 页）
3. **无 shortcode 错误** — 特别注意 `{{< hint >}}` 会导致构建失败

### Phase 5: Git 提交与推送（仅在用户明确要求时执行）

```bash
cd "/Users/pwndazhang/Library/Mobile Documents/com~apple~CloudDocs/6 开发项目/个人主页/x7peeps.github.io"
git add -A
git commit -m "feat: 新增 AI 文章 - <文章标题>"
git push origin master
```

> **⚠️ 经验教训**：写完文章后**不要自动推送**，除非用户明确说"推送"、"提交到 git"、"发布"。用户之前反馈过"没推送到git？"的问题，说明推送这一步很重要但需要用户确认。

## 四、Frontmatter 规范

### 4.1 文章 frontmatter（YAML 格式）

```yaml
---
title: "文章标题：副标题"
weight: 1
tags: [关键词1, 关键词2, 关键词3]
menu:
  main:
    parent: "所属分类名称"
---
```

### 4.2 字段详解

| 字段 | 必填 | 格式 | 说明 |
|------|------|------|------|
| `title` | ✅ | 字符串 | 中文标题，支持冒号分隔主副标题 |
| `weight` | ✅ | 数字 | 同分类内排序权重，数字越小越靠前 |
| `tags` | ✅ | 数组 | 3-6 个技术关键词，中英文混用 |
| `menu.main.parent` | ✅ | 字符串 | 父级分类名称，**不含数字编号前缀** |

> **注意**：AI 文章**不使用** `date`、`draft`、`categories`、`description` 字段（与安全板块不同）。

### 4.3 Frontmatter 示例

```yaml
---
title: "RAG 技术栈全景：从索引到检索到生成的完整链路"
weight: 1
tags: [RAG, 向量检索, Embedding, LLM, 知识库]
menu:
  main:
    parent: "RAG与知识库"
---
```

```yaml
---
title: "AI Agent 架构全景：ReAct、Plan-and-Execute、Reflexion、LATS"
weight: 1
tags: [AI Agent, ReAct, Plan-and-Execute, Reflexion, LATS]
menu:
  main:
    parent: "Agent架构与框架生态"
---
```

## 五、写作规范

### 5.1 标题公式

```
核心技术主题 + 中文冒号（：）+ 范围/方法/副标题
```

**标题风格示例**：
- ✅ "大语言模型技术栈：从 Transformer 到 GPT/Claude 的架构演进"
- ✅ "RAG 技术栈全景：从索引到检索到生成的完整链路"
- ✅ "AI Agent 架构全景：ReAct、Plan-and-Execute、Reflexion、LATS"
- ✅ "如何训练一个原生 Computer Use 模型"（疑问式标题也可以）
- ✅ "安全知识库 RAG 系统：架构设计、技术选型与实现"

### 5.2 文章正文结构

文章正文以一级标题（`#`）重复 frontmatter 的 title，然后开篇：

```markdown
# 文章标题

开篇段落（2-3 段）...

---

## 1. 第一个大章节
### 1.1 小节
### 1.2 小节

---

## 2. 第二个大章节
### 2.1 小节

---

## N. 总结与展望

## 参考资源
```

### 5.3 开篇公式（2-3 段）

**模式一：行业背景 + 本文定位**（最常用）

```
第 1 段：技术演进背景，说明这个领域为什么重要
第 2 段：当前痛点/认知缺口，现有方案的不足
第 3 段：本文定位 — "本文将从...出发，系统梳理..." / "本文不讨论...，而是..."
```

示例：
> "大语言模型（LLM）已从学术论文中的实验性工作，演变为支撑数十亿用户日常交互的基础设施级产品。作为 AI 应用开发者，理解从底层 Transformer 架构到上层应用框架的完整技术栈，是做出正确技术选型、构建高质量 AI 应用的前提。"
>
> "本文不讨论'什么是 AI'这类入门概念，而是以技术栈纵览的方式，系统梳理..."

**模式二：问题驱动 + 痛点共鸣**

```
第 1 段：具体场景中的痛点描述
第 2 段：为什么现有方案不够
第 3 段：本文要解决什么
```

示例：
> "安全团队的知识管理长期面临'三难困境'——文档分散导致检索低效、经验依赖导致知识流失、回答不一致导致决策风险。"

### 5.4 章节编号规则

AI 技术文章使用**阿拉伯数字编号**：

```
## 1. 基础概念 / 技术全景
## 2. 核心原理 / 架构设计
## 3. 关键技术详解
## 4. 对比分析
## 5. 工程实践
## 6. 总结与展望
## 参考资源
```

> **注意**：安全板块使用 `0x00`、`0x01` 十六进制编号，AI 板块使用 `1.`、`2.` 阿拉伯数字编号，**不要混用**。

### 5.5 章节内部结构

每个章节内部遵循：
```
概念定义 → 原理/机制 → 代码/配置示例 → 适用场景/对比 → 小结
```

小节编号：`### 1.1`、`### 1.2`、`### 2.1` ...

### 5.6 内容深度要求

| 维度 | 要求 |
|------|------|
| **字数** | 2500-6000 中文字（深度文章可达 8000+） |
| **目标受众** | 有基础的技术从业者，非纯入门级 |
| **深度定位** | 系统性梳理 + 工程实践视角，强调"为什么"而非仅"怎么做" |
| **实战导向** | 每篇文章包含可直接运行的代码/配置/命令 |
| **可视化节奏** | 每 300-500 字插入一个可视化元素 |

### 5.7 语言风格

1. **中英混排**：技术术语保持英文原文，解释用中文
   - ✅ "Self-Attention 的核心操作是..."
   - ✅ "In-Context Learning 的诞生"
   - ❌ 首次出现时翻译为中文术语

2. **加粗强调**：核心概念、关键结论、重要警告使用 `**加粗**`
   - "**对序列中的每个 Token，动态计算它与所有其他 Token 的相关性权重**"

3. **形象比喻**：善用比喻降低理解门槛
   - "散列函数是密码学的'指纹'提取器"
   - "注意力分数本质上是一个**可微分的软路由机制**"

4. **直白警告**：对常见误区给出直接、不留情面的警告
   - "过度设计一个简单任务的 Agent 架构，往往比选择简单方案更糟糕"

5. **引用块**：用于实战提示、注意事项
   - `> 日常接触的 XX 场景中...`

6. **列表枚举**：技术要点使用有序/无序列表
   - "**核心创新**：首次大规模验证了..."
   - "**架构**：12 层 Transformer Decoder，117M 参数"

### 5.8 可视化元素

保持阅读节奏，每 300-500 字插入一个：

1. **对比表格**：技术对比、属性列举、方案选型

```markdown
| 方案 | 代表模型 | 原理 | 特点 |
| :--- | :--- | :--- | :--- |
| **RoPE** | LLaMA、Qwen | 通过旋转矩阵编码位置 | 当前主流 |
| **ALiBi** | BLOOM、MPT | 对 Attention 分数加偏置 | 实现极简 |
```

2. **代码块**：Python 为主，语言标记正确

```markdown
```python
# 伪代码 / 框架代码
def forward(self, x):
    q, k, v = self.wq(x), self.wk(x), self.wv(x)
    return self.attention(q, k, v)
```　
```

3. **ASCII 架构图**：展示系统架构、数据流、核心循环

```markdown
```
┌──────────────────────────────────────────────┐
│              Agent 核心循环                    │
│   ┌──────────┐                               │
│   │ Perceive │ ← 接收用户输入或环境反馈         │
│   └────┬─────┘                               │
│        ▼                                     │
│   ┌──────────┐                               │
│   │  Think   │ ← 推理当前状态，规划下一步       │
│   └────┬─────┘                               │
│        ▼                                     │
│   ┌──────────┐                               │
│   │   Act    │ ← 执行工具调用或生成响应         │
│   └──────────┘                               │
└──────────────────────────────────────────────┘
```　
```

4. **引用块**（`>`）：实战提示、注意事项、关键洞察

5. **分隔线**（`---`）：大章节之间用水平线分隔

## 六、Shortcode 规范

### 6.1 可用 Shortcode

| Shortcode | 用途 | 适用位置 |
|-----------|------|----------|
| `{{< article_cards >}}` | 展示子文章卡片列表 | **仅**用于分类入口的 `_index.md` |
| `{{< notice >}}` | 提示信息框 | 文章正文内 |

### 6.2 禁止使用的 Shortcode

| Shortcode | 状态 | 说明 |
|-----------|------|------|
| `{{< hint >}}` | ❌ **禁止** | 站点未安装此 shortcode，**会导致 Hugo 构建失败** |

> **⚠️ 血泪教训**：`{{< hint >}}` 会导致构建报错 `Error: module "github.com/McShelby/hugo-theme-relearn" not found`，实际原因是 shortcode 不存在。必须使用 `{{< notice >}}` 替代。如果发现已有的文章使用了 `{{< hint >}}`，需要修复：
> ```bash
> find content/ -name "*.md" -exec sed -i '' 's/{{< hint /{{< notice /g; s/{{< \/hint >/{{< \/notice >/g' {} +
> ```

## 七、批量写作指南

当用户要求一次写多篇文章时：

### 7.1 分批策略

- 按主题领域分批（P1/P2/P3），每批 8-13 篇
- 每批使用 `Task` 工具的 `general_purpose_task` 并行执行
- 每个子任务包含：文章标题、所属分类、调研关键词、大纲要求

### 7.2 并行执行模式

```
主 Agent：制定总计划 → 拆分为 P1/P2/P3
   ├── P1（子 Agent）：写 01-LLM、02-Prompt、03-Agent 文章
   ├── P2（子 Agent）：写 04-RAG、05-评测、06-工程化 文章
   └── P3（子 Agent）：写 07-工具链、08-安全、09-实战、10-模型训练 文章
```

### 7.3 批量写作注意事项

1. **frontmatter 中的 menu.main.parent 必须匹配实际分类名称**（不含编号）
2. **每个分类下的 weight 值不能冲突** — 先检查已有文章的 weight
3. **分类入口的 `_index.md` 不能被覆盖** — 保持 `{{< article_cards >}}` 不变
4. **写完后统一构建验证** — 一次性 `hugo` 检查所有文章
5. **不要自动推送** — 批量写完后等用户确认再 git push

## 八、新建分类指南

当 AI 板块需要新增分类时：

### 8.1 操作步骤

1. 确定编号（查看已有最大编号 +1）
2. 创建目录：`hugo-src/content/AI/NN-新分类名称/`
3. 创建分类入口 `_index.md`：

```markdown
---
title: "新分类名称"
weight: 1
---
{{< article_cards >}}
```

4. 确保 menu.main.parent 值与目录名去掉编号前缀后的名称一致
5. 构建验证

### 8.2 目录命名规则

- 格式：`NN-中文分类名称`
- 编号：两位数字，从 01 开始递增
- 名称：简洁明了，4-8 个汉字

## 九、常见陷阱与修复

| 问题 | 原因 | 修复方法 |
|------|------|----------|
| Hugo 构建失败 `shortcode not found` | 使用了 `{{< hint >}}` | 替换为 `{{< notice >}}` |
| 文章在侧边栏不显示 | `menu.main.parent` 值含编号前缀 | 去掉 `"01-"` 等前缀 |
| 文章卡片页为空 | 分类入口 `_index.md` 被覆盖 | 确保包含 `{{< article_cards >}}` |
| 文章排序混乱 | weight 值与已有文章冲突 | 查看同分类已有 weight，选择不冲突的值 |
| Hugo 构建报 module not found | 在根目录执行 hugo | 切换到 `hugo-src/` 目录执行 |
| 文章图片不显示 | 使用了本地相对路径 | 使用完整 URL 链接 |
| git push 后页面没更新 | 未推送到远程 | 确认执行了 `git push origin master` |
