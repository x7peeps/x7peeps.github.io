## 二、取证分析深度文章规范

### 2.1 基本要求
- **字符数**：≥ 20,000（用 `wc -m` 验证）
- **章节数**：10+ 章（0x01 到 0x0A 或更多）
- **面向读者**：专业安全人员 / 应急响应人员，内容面向成人读者
- **代码注释**：不要添加任何代码注释

### 2.2 Hugo Front Matter 格式

```yaml
---
title: "{中文标题}"
date: {YYYY-MM-DD}T{HH:MM:SS}+08:00
draft: false
weight: {递增值}
description: "{一句话描述，覆盖所有关键词}"
categories: ["应急响应", "取证分析"]
tags: ["{标签1}", "{标签2}", ... "{标签N}"]
---
```

**weight 递增规则**：每篇文章 weight 递增 10，当前序列：
- 360: 恶意软件行为分析 → 370: WMI持久化 → ... → 900: 大语言模型LLM应用安全
- 当前最高 weight: 930（数字身份与访问管理安全取证）
- 下一篇应为 **940**

**description 规范**：用一句话概括全文核心内容，用中文逗号分隔各个技术点，字数 80-150 字。

**tags 规范**：8-10 个标签，中英文混合，覆盖技术点、工具名、攻击类型、平台名。

### 2.3 章节结构模板

每篇文章必须包含以下标准章节（可根据主题增减）：

| 章节编号 | 必选/可选 | 内容 |
|---------|----------|------|
| 0x01 | **必选** | 技术基础与取证概述（架构、与传统取证差异、工具链） |
| 0x02-0x0N | **必选** | 核心技术章节（每个技术点独立成章，深入展开） |
| 0x0N+1 | **必选** | 证据强度分层与案例关联 |
| 0x0N+2 | **必选** | 自动化检测与狩猎（Sigma/Bash/Python/YARA） |
| 0x0N+3 | **必选** | 公开案例分析（≥ 2 个案例） |
| 最后一章 | **必选** | 参考资料（≥ 8 条，含 URL） |

### 2.4 内容要素（每篇必含）

#### A. 证据强度分层
每篇文章必须包含证据强度分类章节，使用三级分类：
- **🔴 确认恶意**：有明确恶意意图和行为的证据
- **🟡 高度可疑**：强烈暗示恶意活动但需进一步验证
- **🟢 需要关注**：可能为正常行为但需结合上下文判断

#### B. 自动化检测脚本
每篇至少包含以下类型的脚本/规则（各至少 1 条）：
- **Sigma YAML 规则**：适用于 SIEM 平台的日志检测规则
- **Bash 脚本**：Linux/macOS 环境下的自动化狩猎脚本
- **Python 脚本**：数据分析、日志解析、自动化检测
- **YARA 规则**（可选）：恶意软件特征匹配

#### C. 公开案例分析
- ≥ 2 个真实 APT 组织或公开攻击事件案例
- 每个案例包含：攻击链描述、取证发现、IOC、经验教训
- 案例来源：公开威胁情报报告、安全厂商分析报告

#### D. 参考资料
- ≥ 8 条参考资料
- 包含 URL 链接
- 覆盖：官方文档、安全研究论文、开源工具文档、威胁情报报告

### 2.5 写作风格规范
- 中文撰写，**技术术语保留英文**（如：Process Hollowing、DLL Injection、LOLBin）
- 每个技术点都要有具体的**命令/代码示例**
- 使用**表格对比分析**（至少每章 1 个表格）
- 使用**代码块**展示命令、脚本、配置
- 使用 **MITRE ATT&CK 技术编号**标注攻击技术（如 T1562.001）
- 使用**层级标题**组织内容（h2 章节标题 + h3 小节标题）

## 三、漏洞分析专题文章规范

### 3.1 基本要求
- **字符数**：≥ 8,000（产品线专题，用 `wc -m` 验证）
- **覆盖漏洞数**：每篇至少覆盖 3-6 个 CVE/漏洞
- **面向读者**：渗透测试人员 / 安全研究员

### 3.2 Hugo Front Matter 格式

```yaml
---
title: "{产品线名称}高危攻击链专题"
date: {YYYY-MM-DD}T{HH:MM:SS}+08:00
draft: false
weight: {按目录内已有 weight 最大值 +10}
description: "围绕 {产品线} 的高危漏洞攻击链专题分析..."
categories: ["渗透测试", "漏洞分析"]
tags: ["{产品线}", "{CVE编号}", ...]
---
```

### 3.3 章节结构模板

| 章节 | 内容 |
|------|------|
| 0x01 | 产品概述与攻击面分析（架构、部署方式、常见配置问题） |
| 0x02-0x0N | 单个 CVE/漏洞深度分析（每个漏洞独立成章） |
| 0x0N+1 | 攻击链组合利用（多漏洞串联利用场景） |
| 0x0N+2 | 自动化检测（Nuclei YAML / Python PoC / Sigma） |
| 0x0N+3 | 防御建议与修复方案 |
| 最后一章 | 参考资料（≥ 6 条） |

### 3.4 每个 CVE 章节标准内容

```markdown
## 0x0N CVE-XXXX-XXXXX 漏洞名称

### 漏洞概述
- 影响产品与版本
- 漏洞类型（CWE 分类）
- CVSS 评分
- 利用条件

### 技术原理
- 根因分析
- 攻击面定位
- 代码层面漏洞（如适用）

### 复现与 PoC
（HTTP 请求 PoC / Python 脚本 / Nuclei 模板）

### 检测与狩猎
（Sigma 规则 / 日志特征 / 流量特征）

### 修复建议
- 官方补丁
- 临时缓解措施
- 配置加固

### 参考链接
```

### 3.5 PoC 代码规范
- HTTP 请求使用原始格式（不依赖特定工具）
- Python 脚本使用 `requests` 库（项目已使用）
- Nuclei YAML 模板遵循标准格式
- 所有 PoC 仅用于**授权测试**

### 3.6 覆盖清单维护
每新增一个产品线专题后，必须同步更新 `常见高危RCE产品线清单.md`，追加新条目。

## 四、Spec 驱动开发流程

### 4.1 批次规划

每次编写文章采用**批次制**，每批 3 篇文章（并行编写提高效率）：

```
.trae/specs/
  forensics-articles-batch-{N}/
    spec.md      ← 规划文档（为什么写、写什么、验收标准）
    tasks.md     ← 任务列表（可勾选的子任务）
    checklist.md ← 验证清单（字符数、构建、内容要素）
```

### 4.2 选题流程
1. 列出已有文章，排除重复主题
2. 选择 3 个尚未覆盖的高价值领域
3. 创建 spec 目录和文件
4. 并行启动子代理编写

### 4.3 编写流程
1. 创建 spec 文档（spec.md / tasks.md / checklist.md）
2. 并行启动 3 个子代理，每个负责 1 篇文章
3. 每个子代理完成后用 `wc -m` 验证字符数
4. 若字符不足 20,000，子代理自行扩充直到达标
5. 运行 `hugo --minify` 验证构建通过
6. 更新 tasks.md（勾选已完成任务）和 checklist.md
7. **自动执行 Git 提交与推送**（见第七章）

### 4.4 验证检查点
- [ ] 文件已创建且字符数 ≥ 要求
- [ ] Hugo 构建通过（无报错）
- [ ] 包含证据强度分层章节
- [ ] 包含自动化检测脚本
- [ ] 包含公开案例分析（≥ 2 个）
- [ ] 包含参考资料（≥ 8 条）

## 五、Hugo 构建注意事项

### 5.1 已知问题
- AI 目录下的 `hint` shortcode 缺失会导致全局构建报错，这是**预存在的问题**，与新文章无关
- 新文章验证时，只要文章所在页面能正常构建即可
- 构建命令：`cd hugo-src && hugo --minify`

### 5.2 构建验证方法
```bash
cd hugo-src && hugo --minify 2>&1 | tail -15
```
- 确认 `Total in X ms` 输出且无 ERROR
- WARN 级别的 deprecated 提示可忽略

### 5.3 字符数验证
```bash
wc -m "hugo-src/content/安全/应急响应/0x03取证分析/{文件名}.md"
```

## 七、自动 Git 提交与推送

文章编写完成并通过验证后，**必须自动执行 Git 提交与推送**，无需用户额外确认。

### 7.1 提交时机
在以下所有条件均满足后立即执行：
- 所有文章字符数验证通过（≥ 20,000 / ≥ 8,000）
- Hugo 构建通过（`hugo --minify` 无 ERROR）
- 内容要素检查通过（证据分层、脚本、案例、参考资料）

### 7.2 提交步骤

```bash
cd /Users/pwndazhang/Library/Mobile Documents/com~apple~CloudDocs/6 开发项目/个人主页/x7peeps.github.io

git add "hugo-src/content/安全/应急响应/0x03取证分析/{新文章}.md" \
        ".trae/specs/forensics-articles-batch-{N}/"

git commit -m "feat(blue-team): 新增{N}篇蓝队取证分析深度文章 (batch-{N})

- {文章1标题} (weight: {W1}, {字符数} chars)
  {一句话概述}
- {文章2标题} (weight: {W2}, {字符数} chars)
  {一句话概述}
- {文章3标题} (weight: {W3}, {字符数} chars)
  {一句话概述}"

git push
```

### 7.3 Commit Message 规范
- 类型：`feat(blue-team)` 用于新文章，`fix(blue-team)` 用于修正
- 标题格式：`feat(blue-team): 新增{N}篇蓝队取证分析深度文章 (batch-{序号})`
- 正文：每篇文章一行，包含标题、weight、字符数、核心内容概述
- 必须包含所有新增/修改的文件路径

### 7.4 推送验证
- 确认 `git push` 输出无错误
- 确认远程分支更新（如 `master -> master`）
- 若 push 失败，记录错误信息并提示用户介入

### 7.5 注意事项
- 仅提交与本次文章编写直接相关的文件（新文章 + spec 文件）
- 不要提交不相关的临时文件或构建产物
- 如果存在其他未追踪文件，不要一并提交，除非用户明确要求
